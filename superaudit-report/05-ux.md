# 6. UX funcional

> Auditor: ux-auditor | Fecha: 2026-05-21

## Resumen ejecutivo

BrewTUI-Bar es una TUI (Ink/React) con un modelo de navegacion mediante menu lateral y atajos de teclado bien estructurado: los flujos principales (busqueda, instalacion, upgrade, limpieza, seguridad) siguen patrones consistentes de confirmacion, feedback de progreso y resultado. Las fricciones existentes son de severidad Baja o Media: mensajes de error con texto crudo del CLI de brew, ausencia de pantalla de ayuda contextual, cancelacion bloqueada durante rollback, y un literal en ingles en la vista de compliance. No se detectaron flujos criticos rotos ni perdidas de datos.

## 6.2 Criterios de UX por flujo

### Flujo: Primer arranque — Welcome y onboarding

* **Nombre:** Primer arranque
* **Objetivo del usuario:** Entender que es BrewTUI-Bar y llegar al dashboard principal
* **Inicio del flujo:** `<WelcomeView>` renderizada condicionalmente en `src/app.tsx` cuando `isOnboardingComplete()` devuelve `false`
* **Final esperado:** Usuario llega al dashboard con el menu lateral activo
* **Pasos:**
  1. Usuario lanza `brewtui-bar` por primera vez
  2. Se muestra `WelcomeView` con nombre de producto, version y referencia de teclas globales
  3. Usuario presiona Enter, Espacio o Esc para continuar
  4. `markOnboardingComplete()` escribe flag en disco (en `.finally`, no bloquea si falla)
  5. Se renderiza `AppLayout` con `DashboardView` y menu lateral activo
* **Fricciones detectadas:**
  - No existe mecanismo para volver a ver la pantalla de bienvenida tras el primer arranque
  - El menu lateral esta activo por defecto desde el primer frame, lo que puede sorprender si el usuario no ha leido la referencia de teclas
* **Puntos de abandono:** Ninguno relevante; cualquier tecla continua
* **Mensajes ambiguos:** Ninguno
* **Feedback del sistema suficiente:** Si — la pantalla de bienvenida es clara
* **Tiempo a valor aceptable:** Si — dashboard disponible inmediatamente tras una pulsacion de tecla
* **Recuperacion ante error:** Si — fallo al escribir flag no bloquea el flujo
* **Consistencia con el resto del producto:** Si

### Flujo: Busqueda, detalle e instalacion de paquete

* **Nombre:** Busqueda → Package Info → Instalacion

  1. Usuario escribe consulta (minimo 2 caracteres); hint de minimo mostrado si se intenta con menos

* **Puntos de abandono:** Error de busqueda sin sugerencia de accion alternativa
* **Mensajes ambiguos:** Ninguno en flujo normal
* **Feedback del sistema suficiente:** Si — spinner, progress log, result banner
* **Tiempo a valor aceptable:** Si
* **Recuperacion ante error:** Parcial — hay `ResultBanner` de error pero sin hint de "reintentar"

  - No hay hint de donde comprar la licencia si la activacion falla por clave invalida
  - El flujo de revalidacion (`brewtui-bar revalidate`) tiene estados distintos (grace, expired) bien comunicados
* **Puntos de abandono:** Error de red sin instruccion de reintento
* **Mensajes ambiguos:** Posibles mensajes raw de Polar en caso de error inesperado
* **Feedback del sistema suficiente:** Parcial

### Flujo: Sincronizacion iCloud

* **Nombre:** Sincronizacion entre maquinas via iCloud

  1. `initialize(isPro)` lee configuracion existente (solo lectura, sin escritura en iCloud)

* **Mensajes ambiguos:** Falta indicacion de cuantos conflictos quedan para poder continuar
* **Feedback del sistema suficiente:** Parcial

## Analisis transversal

### Consistencia de patrones

**Conforme:** Todos los flujos con operaciones de stream (install, upgrade, cleanup, rollback, brewfile reconcile) usan `ProgressLog` + `ResultBanner`. Los dialogs de confirmacion destructiva usan `ConfirmDialog` con el mismo patron `Y/N`. Los errores de carga usan `ErrorMessage` con prefijo `error_prefix`.

**Parcial:** El feedback de exito tras operaciones sin stream (crear perfil, editar perfil) no sigue el patron `ResultBanner` del resto del producto.

### Manejo de errores

**Conforme:** Estados de error de Zustand stores se muestran en las vistas (`if (error) return <ErrorMessage message={error} />`). Rate limiting en activacion Pro correctamente traducido. Errores de red en security audit detectados via `isNetworkError()`.

**No conforme:** `err.message` raw de CLI de Homebrew puede aparecer en `SearchView` y `PackageInfoView` sin filtrado ni traduccion.

### Patrones de navegacion

**Conforme:** `useViewInput` asegura que el menu lateral suprime todos los handlers de vista de forma consistente. `goBack()` cierra el menu si `menuMode` esta activo antes de navegar atras. Esc tiene comportamiento consistente: cierra modales, cancela streams, vuelve a la vista anterior.

**Parcial:** En `ProfilesView`, Esc en sub-pasos de creacion/edicion salta al listado en lugar de retroceder un paso en el flujo.

### Mecanismos de feedback

**Conforme:** Exito e error en streams siempre comunicados con `ResultBanner`. Loading states con `Loading` (spinner + label descriptivo). Cache age en security audit via `formatRelativeTime`.

**No conforme:** Crear/editar perfil no muestra ningun feedback de exito.

### Calidad del copy

**Conforme:** Todas las cadenas de usuario pasan por `t()` con claves tipadas; `npm run typecheck` detecta claves faltantes. Mensajes de confirmacion incluyen datos concretos (nombres de paquetes, conteos). Soporte de plurales via `tp()`.

**No conforme:** `(errors)` y `(warnings)` en `compliance.tsx:80,91` estan fuera del sistema de traduccion.
